# 미팅 에이전트 요구사항 정의 및 설계 통합본 v2.0

2026년 4월 | Parametacorp

> **v1.0 → v2.0**: v1 기반 구현 후 실제 운영에서 발견된 이슈를 진단하고, 데이터 구조·품질관리·동시성·제안서 워크플로우 등 신규 요구사항을 추가한 개정판입니다.

---

## 0. v1 → v2 변경 요약

| 영역 | v1 설계 | v2 변경 | 변경 이유 |
|------|---------|---------|-----------|
| Drive 구조 | 플랫 파일 (`Companies/카카오.md`) | Wiki 구조 + `[[링크]]` + 출처 섹션 | 자료 간 연결 없이 고립되어 맥락 추적 불가 |
| 검색·색인 | 없음 (파일명 매칭만) | SQLite 메타데이터 인덱스 + Drive 전문검색 | 최근 10개만 조회 가능, 업체·기간 필터 불가 |
| 산출물 품질 | LLM 1회 생성 → 즉시 발송 | 필수항목 검증 + 자동 재생성 루프 | 누락된 섹션이 걸러지지 않음 |
| 제안서 | v1 §4.3에 명시되었으나 미구현 | 에이전트 제안형 워크플로우 (개요 확인 → 생성 → 수정) | 미팅 맥락 기반 intake + 사용자 확인 필수 |
| 브리핑 업체 인식 | `extendedProperties`에 업체명 있을 때만 리서치 | 미팅 제목 LLM 추론 + 기존 업체 목록 매칭 폴백 | 수동 생성·외부 초대 일정에서 리서치 누락 |
| 동시성 | 고려 없음 (1인용 설계) | 공유 상태 Lock + event_id 기반 키 전환 | 복수 미팅 동시 종료 시 데이터 유실·덮어쓰기 |
| Trello 코멘트 | 체크리스트만 명시 | 회의 요약 코멘트 자동 추가 | 체크리스트만으로는 "무슨 맥락이었는지" 안 남음 |
| 원본 보관 | 가공본만 저장 | 원본(트랜스크립트·이메일)도 별도 보관 | 재가공·감사 추적 불가 |
| 회의록 초안 | 메모리에만 보관 | 파일 영속화 (서버 재시작 시 복구) | 서버 재시작 시 초안 유실 |

## 1. 운영 이슈 진단

v1 기반 구현체를 운영하며 발견된 이슈 10건을 진단합니다.

### 이슈 1 — 산출물 목록과 저장 체계

**현황**: 6종 산출물(내부/외부 회의록, 액션아이템, Slack DM, Gmail, Trello)이 생성되나, 산출물 간 연결이 없고 "이 회의록이 어떤 정보를 참고해서 만들어졌는지" 추적 불가.

**문제점**:
- 회의록 파일에 참석자·업체 정보가 텍스트로만 존재 → 역추적 불가
- 원본 트랜스크립트가 보관되지 않아 재가공 불가
- 산출물 간 상호 참조 없음 (회의록 → 업체 파일, 업체 파일 → 회의록)

### 이슈 2 — Drive 파일구조와 자료 관리

**현황**: `Contacts/Companies/`, `Contacts/People/`, `Minutes/` 3개 폴더에 플랫 파일 저장.

**문제점**:
- 파일끼리 연결이 없음 — "홍길동이 참석한 회의 목록"을 알 수 없음
- 출처 정보 없음 — 기업 파일의 각 항목이 웹 검색인지, Gmail인지, ParaScope인지 불명
- 원본 문서 미보관 — 가공 요약만 저장되어 원본 확인 불가

### 이슈 3 — 검색·색인 부재

**현황**: `/회의록` 명령으로 최근 10개 파일명 목록만 제공. 브리핑 시 `get_previous_context()`가 파일명에 업체명이 포함된 회의록만 매칭.

**문제점**:
- 업체명·기간·키워드 기반 검색 불가
- 10건 넘으면 과거 회의록 접근 불가
- 미팅 제목에 업체명이 없으면 이전 맥락 연결 실패

### 이슈 4 — 산출물 품질 검증 부재

**현황**: 회의록 템플릿(내부용/외부용)과 작성 규칙(6가지)은 존재. 그러나 생성된 결과물에 대한 자동 검증이 없음.

**문제점**:
- LLM이 필수 섹션(액션아이템, 주요 결정)을 누락해도 사용자에게 그대로 전달
- 액션아이템 테이블 형식 오류(파이프 누락 등) 검출 안 됨
- 브리핑 리서치 결과는 검증 없이 즉시 Slack 발송

### 이슈 5 — 제안서·내부미팅 패키지 미구현

**현황**: v1 §4.3에서 "내부 미팅 패키지(요약 브리핑, 리서치 초안, 제안서 초안, 의사결정 항목)"를 명시했으나 미구현 상태(FR-A06, FR-A07 ⬜).

**문제점**:
- 제안서 같은 절차적 산출물은 자동 생성하면 맥락 없는 결과물이 나옴
- 사용자 확인 없이 생성하면 불필요한 산출물 발생
- intake(목적, 대상, 범위) 설정 없이 품질 보장 불가

### 이슈 6 — 브리핑 시 업체 인식 실패

**현황**: 캘린더 이벤트의 `extendedProperties.private.company`에 업체명이 저장된 경우에만 리서치가 실행됨. 없으면 `_send_internal_briefing()`으로 헤더만 발송.

**문제점**:
- 에이전트가 생성하지 않은 일정(수동 생성, 외부 초대)에는 `company` 속성이 없음
- `after.py`에 `_infer_company_name()` 함수가 있으나 `before.py` 브리핑에서는 미사용
- 미팅 제목이 "홍길동 상무 미팅"처럼 인물명만 있으면 추론 실패

### 이슈 7 — 기업정보 수동 입력 경로 부재

**현황**: 기업 정보는 자동 웹 검색 또는 `/company` 명령으로만 수집. 명함 OCR은 인물 정보만 처리.

**문제점**:
- 사용자가 직접 알고 있는 기업 정보를 텍스트로 입력하는 경로가 없음
- 비공개 정보(내부 평판, 담당자 성향 등)를 기록할 방법이 없음

### 이슈 8 — 동시성 문제

**현황**: 모든 공유 상태(`_active_sessions`, `_pending_minutes`, `_pending_agenda` 등)가 모듈 레벨 딕셔너리로 관리되며, `threading.Lock`이 전혀 없음.

**문제점**:
- 같은 사용자의 두 미팅이 동시에 종료되면 `_pending_minutes[user_id]` 덮어쓰기 발생
- 스케줄러 스레드(10분 주기)와 이벤트 핸들러 스레드가 같은 딕셔너리를 동시 접근
- JSON 파일(`pending_agenda.json`, `meeting_drafts.json`) 동시 쓰기 시 파일 손상 가능
- `_pending_minutes`가 영속화되지 않아 서버 재시작 시 검토 대기 중인 초안 유실

### 이슈 9 — Trello 코멘트 미활용 (dead code)

**현황**: `trello.py:305-322`에 `add_comment()` 함수가 구현되어 있으나, 프로덕션 코드 어디에서도 호출하지 않음.

**문제점**:
- Trello 카드에 체크리스트만 달리고, 회의 맥락이 남지 않음
- 시간이 지나면 "이 체크리스트가 왜 생겼는지" 추적 불가

### 이슈 10 — Trello 등록 양식

**현황**: 체크리스트 항목은 `[담당자] 내용 (기한: YYYY-MM-DD)` 형식으로 통일. 사용자 승인 후 등록(자동 등록 아님). 단, 카드가 없으면 확인 없이 자동 생성.

**문제점**:
- 코멘트 양식이 정의되지 않음 (이슈 9와 연결)
- 카드 자동 생성 시 사용자 확인이 없어 의도치 않은 카드 생성 가능

회의록 파일 하단에 관련 자료 링크를 자동 삽입:

```markdown
## 관련 자료
- 업체: [[카카오]]
- 참석자: [[홍길동]], [[김영희]]
```

#### 2.1.5 구현 방식

- **`[[링크]]` 삽입 시점**: `save_minutes()`, `save_company_info()`, `save_person_info()` 호출 시 자동
- **역링크 갱신**: 회의록 저장 시 → 해당 업체/인물 파일의 `## 미팅 히스토리` 테이블에 행 추가
- **출처 태깅**: `research_company()`에서 정보 수집 시 `[출처: {type}]` 인라인 태그 부착
- **원본 보관**: `Sources/` 폴더에 트랜스크립트 원문, 이메일 스레드 원문, 웹 검색 결과 원문 저장

### 2.2 검색·색인 시스템

#### 2.2.1 SQLite 메타데이터 인덱스

`store/users.db`에 테이블 추가:

```sql
CREATE TABLE meeting_index (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id      TEXT NOT NULL,
    user_id       TEXT NOT NULL,
    date          TEXT NOT NULL,          -- YYYY-MM-DD
    title         TEXT NOT NULL,
    company_name  TEXT,                   -- 복수 업체는 쉼표 구분
    attendees     TEXT,                   -- JSON 배열
    drive_file_id TEXT,                   -- 내부용 회의록 Drive ID
    drive_link    TEXT,                   -- 웹 링크
    has_proposal  INTEGER DEFAULT 0,      -- 제안서 존재 여부
    created_at    TEXT DEFAULT (datetime('now'))
);

CREATE INDEX idx_meeting_company ON meeting_index(company_name);
CREATE INDEX idx_meeting_date ON meeting_index(date);
CREATE INDEX idx_meeting_user ON meeting_index(user_id);
```

#### 2.2.2 검색 명령어

| 명령어 | 동작 | 예시 |
|--------|------|------|
| `/회의록 {업체명}` | 해당 업체 회의록 전체 목록 | `/회의록 카카오` |
| `/회의록 {YYYY-MM}` | 해당 월 회의록 목록 | `/회의록 2026-03` |
| `/회의록 {업체명} {기간}` | 업체 + 기간 필터 | `/회의록 카카오 2026-03` |
| `{업체명} 회의록 찾아줘` | 자연어 검색 | `카카오 지난달 회의록 찾아줘` |

#### 2.2.3 구현 시점

- 회의록 Drive 저장 성공 후 → `meeting_index` INSERT
- 기존 회의록은 마이그레이션 스크립트로 일괄 인덱싱

### 2.3 산출물 품질 검증

#### 2.3.1 필수항목 검증 (회의록)

회의록 초안을 사용자에게 보여주기 전에 자동 검증:

```
검증 항목:
├── 내부용 회의록
│   ├── [필수] ## 회의 요약 섹션 존재
│   ├── [필수] ## 액션 아이템 섹션 + 테이블 형식
│   ├── [필수] ## 주요 결정 사항 섹션 존재
│   ├── [필수] ## 주요 논의 내용 섹션 존재
│   ├── [권장] 본문 500자 이상
│   └── [권장] 액션아이템 1건 이상 (경고만)
│
└── 외부용 회의록
    ├── [필수] ## 회의 개요 섹션 존재
    ├── [필수] ## 주요 합의 사항 섹션 존재
    ├── [필수] ## 공동 액션 아이템 섹션 존재
    ├── [금지] "내부 메모" / "협상" / "전략" 키워드 없음
    └── [권장] 내부용보다 짧음
```

**처리 흐름**:
1. LLM이 초안 생성
2. `validate_minutes(body, type)` 실행
3. 필수항목 누락 시 → LLM에게 "다음 섹션이 누락되었습니다: {목록}. 보완해서 다시 작성해주세요." 자동 재생성 (최대 2회)
4. 금지 키워드 검출 시(외부용) → 해당 부분 자동 제거 후 재구성
5. 검증 통과 후 사용자에게 제시

#### 2.3.2 브리핑 품질 검증

브리핑은 실시간성이 중요하므로 경량 검증만:

```
├── 기업 뉴스: 항목 1건 이상, URL 포함 여부
├── 인물 정보: 이름/직책 최소 포함
├── 서비스 연결점: 1건 이상
└── 이전 맥락: 존재 시 누락 없이 포함
```

누락 시 재생성이 아닌 "⚠️ {항목} 정보를 찾지 못했습니다" 표시.

### 2.6 동시성 안정화

#### 2.6.1 Lock 적용

모든 공유 딕셔너리에 `threading.Lock` 적용:

```python
# during.py
_sessions_lock = threading.Lock()
_minutes_lock = threading.Lock()
_inputs_lock = threading.Lock()

# before.py
_agenda_lock = threading.Lock()
_drafts_lock = threading.Lock()
```

#### 2.6.2 키 전환: user_id → event_id

`_pending_minutes`의 키를 `user_id`에서 `event_id`로 변경하여 동일 사용자의 복수 미팅이 독립적으로 처리되도록 함:

```python
# 변경 전
_pending_minutes[user_id] = draft_data

# 변경 후
_pending_minutes[event_id] = draft_data
# Slack 버튼 value에 event_id 포함하여 매핑
```

#### 2.6.3 대기열 + 알림

같은 사용자가 진행 중인 작업이 있을 때:

```
사용자: 미팅2 종료 요청
에이전트: "미팅1 회의록을 처리 중입니다. 
         완료되면 미팅2도 바로 시작할게요. ⏳"
미팅1 완료 → 미팅2 자동 시작 → 완료 알림
```

#### 2.6.4 회의록 초안 영속화

`_pending_minutes`를 파일로도 백업하여 서버 재시작 시 복구:

```python
# 초안 생성 시
_pending_minutes[event_id] = draft_data
_save_pending_minutes()  # .sessions/pending_minutes.json

# 서버 시작 시
_pending_minutes = _load_pending_minutes()
```
